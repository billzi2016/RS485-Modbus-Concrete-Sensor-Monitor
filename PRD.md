# 产品需求文档（PRD）

## 1. 项目名称

大型水利大坝分布式混凝土应变高并发实时监测系统（跨平台重构版）

---

## 2. 项目背景

本项目来源于一个早期竞赛 / 科研原型。原始版本主要用于演示大坝混凝土内部应变、温度及设备状态的实时采集与展示，技术栈较为简单，界面风格偏传统工控软件，重点在于“能采到、能显示、能看趋势”。

随着项目目标从原型验证转向可持续维护和跨平台运行，现计划对原方案进行一次重构：

- 保留原始业务场景和核心通信模型
- 保留高并发采集、短历史展示、异常状态提示等关键能力
- 放弃平台绑定较强的桌面实现方式
- 改为基于 Python 的跨平台架构，支持 macOS、Windows、Linux 开发与部署

本次重构的目标不是追求复杂架构，而是在实现简单、结构清晰、可维护的前提下，支撑大规模网关并发采集和监控展示。

---

## 3. 项目目标

### 3.1 总体目标

构建一套可跨平台运行的混凝土应变实时监测系统，用于在局域网环境下并发采集 100+ 个 Modbus TCP 网关的数据，并通过 Web 监控界面展示最新状态和最近 10 分钟历史趋势。

### 3.2 核心目标

- 支持 100+ 个网关并发采集
- 支持单网关下 16 个传感器节点的数据解析
- 对采集结果进行纯内存级高速缓存
- 使用 Redis 作为运行时缓存中间层，解耦采集器与 Web 展示层
- 提供传统工控风格的 Web 监控界面
- 展示实时值、状态信息和最近 10 分钟短历史
- 网关离线或测点异常时，前端显示 `Err`

### 3.3 非目标

当前阶段不纳入以下内容：

- 长周期历史归档
- 关系型数据库建模与持久化分析
- 复杂用户权限体系
- 手机端适配
- 三维可视化或花哨大屏展示

---

## 4. 业务场景

系统部署在中控室或实验室局域网内，负责轮询多个工业网关。每个网关下挂接一组 RS-485 传感器，网关对下位串口通信进行封装，对上只暴露 Modbus TCP 接口。上位系统无需处理底层 485 冲突，只需要面向网关进行标准 TCP 读操作。

典型使用方式如下：

- 运维人员打开监控页面查看当前网关在线情况
- 页面显示各测点当前应变、温度、频率、状态码
- 当某个测点通信异常或状态异常时，页面直接显示 `Err`
- 运维人员点击某个测点后，查看其最近 10 分钟趋势曲线

---

## 5. 总体架构

系统采用“采集、缓存、展示”三层结构：

### 5.1 采集层

使用 Python `asyncio` 实现独立运行的异步采集器：

- 并发轮询 100+ 个 Modbus TCP 网关
- 独立处理每个网关的连接、超时、重试与报文解析
- 采集器作为独立进程运行，不直接耦合 Django 请求生命周期

### 5.2 缓存层

使用 Redis 作为运行时缓存：

- 保存每个测点的最新快照
- 保存每个测点最近 10 分钟历史窗口
- 作为采集层与 Web 层之间的数据交换中间层

### 5.3 展示层

使用 Django 构建 Web 应用：

- `Django Template` 负责页面渲染
- `Django Ninja` 负责前端异步查询接口
- 前端使用原生 HTML / CSS / JS 实现高密度监控界面
- 历史趋势图使用轻量图表库展示

## 6. 技术选型

### 6.1 后端

- Python 3.11+
- `asyncio`：高并发采集调度
- Django：Web 项目骨架与页面组织
- Django Ninja：API 接口层
- Redis：运行时缓存

### 6.2 前端

- Django Template
- 原生 HTML / CSS / JavaScript
- Chart.js：用于 10 分钟短历史曲线展示

### 6.3 选型原则

- 优先简单、稳定、容易维护
- 不做前后端分离
- 不引入不必要的重型消息中间件
- 第一阶段不依赖关系型数据库

---

## 7. 网络与设备模型

### 7.1 网络模型

- 中央监控系统运行在工业局域网中
- 网关采用独立 IP 地址接入
- 网络规模为 100+ 个 Modbus TCP 网关
- 网关分布在同一局域网或相近网段中

### 7.2 设备模型

- 每个网关下挂 16 个传感器节点
- 每个节点采集以下指标：
  - 原始振弦频率值
  - 实时受力应变值
  - 实时补偿温度值
  - 历史最大应变峰值
  - 硬件健康状态码

### 7.3 测点标识

系统使用统一测点键作为内部主键：

`GatewayIP_SlaveID_SensorIndex`

示例：

`10.0.1.25_1_12`

## 8. 功能需求

### 8.1 网关配置管理

系统应支持通过配置文件维护网关列表，至少包含：

- 网关 IP
- 端口
- 从站号
- 轮询周期
- 网关名称或分组名称

首阶段配置方式可采用静态文件加载，不要求页面化增删改。

### 8.2 高并发异步采集

采集器需支持以下行为：

- 基于 `asyncio` 为每个网关建立独立采集任务
- 使用批量调度方式并发轮询所有网关
- 某个网关超时、掉线或返回异常时，不影响其他网关
- 支持轮询超时控制和有限次数重试
- 每轮采集后将结果写入 Redis

### 8.3 报文解析

采集器需要对网关返回的固定格式报文进行解析：

- 按大端序解析字节流
- 使用整型位移方式拼装数值
- 不依赖浮点字节序
- 根据约定缩放比例还原真实物理量

### 8.4 最新快照缓存

系统需要缓存每个测点的最新状态，至少包括：

- 实时应变
- 实时温度
- 原始频率
- 历史峰值
- 状态码
- 更新时间
- 在线 / 离线状态

最新快照应可被 Django 页面和接口快速读取。

### 8.5 最近 10 分钟短历史

系统需要保留每个测点最近 10 分钟数据窗口，用于现场观察趋势：

- 默认按采样时间顺序保存
- 仅保留短历史，不做长期归档
- 页面可按测点拉取其最近 10 分钟曲线
- 历史窗口到期后自动覆盖旧数据
- 图表按指标使用固定量程，避免不同测点因自动缩放导致现场误判

短历史图表的默认量程如下：

- 温度：`-10 ~ 40`
- 频率：`1000 ~ 3000`
- 应变：`-5000 ~ 5000`
- 最大值：`-10000 ~ 10000`

### 8.6 监控界面

系统提供传统工控风格的 Web 界面，整体视觉应接近任务管理器或老式监控软件：

- 信息密度高
- 无复杂动画
- 白色背景、灰色边框、小字体
- 无圆角、无卡片化、无大留白
- 以可读性和稳定性优先

界面采用多 Tab 结构，至少包含以下指标页：

- `应变`
- `温度`
- `频率`
- `最大值`

默认进入 `应变` Tab。

页面至少包含以下区域：

- 总览区：在线网关数、离线网关数、异常测点数、最后更新时间
- Tab 区：切换不同指标视图
- 列表区：测点实时数据矩阵
- 详情区：选中测点最近 10 分钟趋势图

列表区布局要求如下：

- 以 `21:9` 宽屏显示器为主要设计目标
- 每一行对应一个网关 IP
- 每一行固定展示该网关下 `16` 个测点
- 左侧显示网关 IP，右侧紧凑排列 16 个测点方框
- 每个测点方框仅显示当前值或 `Err`
- 方框使用小边距、细灰边框、直角样式
- 页面应尽量减少滚动和装饰元素，优先保证单屏密集展示

### 8.7 异常显示

当出现以下情况时，系统应在页面中显示异常状态：

- 网关离线
- 轮询超时
- 报文长度不合法
- 状态码指示硬件异常

对于无法取得有效值的测点：

- Redis 中应写入异常状态标记
- Django 页面不显示数值，直接显示 `Err`

## 9. 接口与缓存设计

### 9.1 采集器写 Redis

采集器将数据写入 Redis，建议分为两类：

1. 最新快照
- 每个测点一个键
- 保存当前最新值与状态

2. 短历史
- 每个测点一个历史序列
- 保存最近 10 分钟采样点

### 9.2 Django 读 Redis

Django 不直接参与采集，只负责读取 Redis 并提供：

- 页面渲染数据
- 前端轮询接口
- 测点详情和历史曲线接口

### 9.3 前端刷新方式

首阶段采用简单可靠的轮询模式：

- 总览和列表数据：1 秒轮询一次
- 历史图：选中测点后按 1 到 2 秒轮询刷新
- Tab 切换后仅刷新当前指标页所需数据

当前阶段不强制要求 WebSocket。

## 10. 协议规范

### 10.1 单传感器载荷布局

每个传感器节点固定使用 14 字节：

| 偏移 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| 0 ~ 1 | 原始振弦频率值 | `UInt16` | 原始频率 |
| 2 ~ 5 | 实时应变值 | `Int32` | 放大 1000 倍 |
| 6 ~ 7 | 实时温度值 | `Int16` | 放大 100 倍 |
| 8 ~ 11 | 历史最大应变峰值 | `Int32` | 放大 1000 倍 |
| 12 ~ 13 | 硬件状态码 | `UInt16` | 状态字 |

### 10.2 单网关报文长度

单网关下 16 个节点合并返回：

- MBAP 报头：7 字节
- 功能码：1 字节
- 字节计数：1 字节
- 数据区：16 × 14 = 224 字节

总长度：

`7 + 1 + 1 + 224 = 233 字节`

### 10.3 解算规则

- 实时应变值：接收 `Int32` 后除以 `1000`
- 实时温度值：接收 `Int16` 后除以 `100`
- 历史峰值：接收 `Int32` 后除以 `1000`
- 所有解析过程使用整型拼装与缩放，不依赖 `float` 传输

## 11. 核心处理流程

### 11.1 数据流

`Modbus TCP 网关 -> asyncio 采集器 -> Redis -> Django/Ninja -> 浏览器`

### 11.2 单轮处理流程

1. 采集器从配置文件读取网关列表
2. 为每个网关创建独立异步任务
3. 周期性发起 Modbus TCP 读取请求
4. 校验返回帧长度和数据区
5. 解析 16 个测点数据
6. 将最新快照写入 Redis
7. 将短历史追加写入 Redis
8. Django 页面通过接口读取并刷新显示

---

## 12. 非功能需求

### 12.1 性能

- 支持 100+ 个网关并发轮询
- 支持 1600+ 测点实时展示
- 整体一轮轮询目标控制在 1.5 秒以内

### 12.2 稳定性

- 单网关异常不影响全局采集
- 页面刷新失败不影响采集进程继续运行
- Redis 短时读写波动不应导致进程整体崩溃

### 12.3 内存与缓存控制

- 历史数据仅保留最近 10 分钟
- 不在本阶段做长期原始报文保存
- 控制 Redis 与采集器内存占用在合理范围内

### 12.4 可维护性

- 采集器与 Web 层职责分离
- 框架和目录结构保持清晰
- 接口和页面不做过度抽象

---

## 13. 异常与容错

### 13.1 网关异常

当网关离线、连接超时或返回非法报文时：

- 记录本轮异常
- 将对应测点标记为不可用
- 页面显示 `Err`

### 13.2 状态码异常

当硬件状态码提示断线、短路或其他自检失败时：

- 测点状态标记为异常
- 页面高亮或直接显示 `Err`

### 13.3 数据边界

对于明显越界或格式异常的数据：

- 丢弃当前值
- 不以异常值覆盖有效缓存
- 记录异常计数，便于后续排查

## 14. 界面要求

### 14.1 风格要求

- 传统 Windows / 工控监控风格
- 不追求现代卡片化设计
- 不使用复杂动画和大面积渐变
- 页面主背景为白色
- 数据单元使用灰色细边框方块
- 所有控件保持直角，不使用圆角
- 强调稳定、清晰、可持续盯屏查看

### 14.2 交互要求

- 默认进入总览页面
- 默认进入 `应变` Tab
- 支持按网关、分组、状态筛选
- 支持按 Tab 切换查看应变、温度、频率、最大值
- 支持点击测点查看详细历史趋势
- 支持快速识别异常设备与异常测点

### 14.3 宽屏布局要求

- 页面优先适配 `21:9` 宽屏工控显示器
- 总览信息区保持单行紧凑排布
- 图表区 `legend` 固定显示在右上角
- 同一指标下所有子图统一量程，不做自动缩放
- IP 标识列保持固定宽度，便于快速横向扫视
- 16 个测点单元格应保持统一尺寸与对齐方式

## 17. 总结

本项目是对既有竞赛 / 科研原型的一次工程化重构。重构后系统不再依赖单一桌面平台，而是采用 `Python + asyncio + Redis + Django + Ninja + Template` 的跨平台方案，实现：

- 高并发采集
- 短历史缓存
- 传统工控风格监控页面
- 简单但可维护的系统结构

该方案优先保证“先稳定采集、稳定展示、便于维护”，再为后续持久化、告警和管理能力扩展留出空间。
